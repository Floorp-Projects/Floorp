# Floorp プロジェクト LLM 向けドキュメント

このディレクトリには、AI アシスタント（LLM）が Floorp プロジェクトを理解し、効果的に支援するためのドキュメントが含まれています。

## 目的

このドキュメント群は以下の目的で作成されています：

1. **プロジェクト理解の加速**: Floorp の全体像を素早く把握
2. **開発支援の質向上**: アーキテクチャを理解した上での適切な提案
3. **知識の保存**: プロジェクトの重要な情報を構造化して保存
4. **コンテキストの提供**: LLM が開発タスクを支援する際の背景情報

## ドキュメント一覧

### 1. [プロジェクト概要](./project-overview.md)

**対象**: 初めて Floorp プロジェクトに触れる開発者や LLM

- 主要コンポーネントの概要
- ビルドシステムの基本
- CI/CD とリリースプロセス
- 国際化とカスタマイズ

**推奨される使い方**:
- プロジェクト全体を理解する最初のステップとして読む
- 新しい機能がどこに配置されるべきか判断する際の参考に
- 技術スタックの選択基準を理解する

### 2. [開発ノート](./development-notes.md)

**対象**: 実際に開発を行う開発者や開発支援を行う LLM

**内容**:
- 開発環境のセットアップ手順
- アーキテクチャ上の重要な決定事項
- よくある問題と解決策

**推奨される使い方**:

### 3. [アーキテクチャ詳細ガイド](./architecture-deep-dive.md)

**対象**: アーキテクチャ理解が必要な開発者や複雑な変更を支援する LLM

**内容**:
- システム全体のアーキテクチャ図

- 主要機能の実装パターン
- 拡張性と保守性の設計

**推奨される使い方**:
- 大規模な機能追加や変更を行う前に読む
- 既存の機能がどのように動作しているか深く理解する
- 新しいアーキテクチャパターンを導入する際の参考に

## ドキュメントの使い方

### シナリオ別ガイド

#### シナリオ 1: 初めて Floorp プロジェクトに触れる

1. **[プロジェクト概要](./project-overview.md)** を読む（20-30分）
2. セットアップ手順に従って開発環境を構築
3. **[開発ノート](./development-notes.md)** のベストプラクティスを確認

#### シナリオ 2: 新しい機能を追加する

1. **[プロジェクト概要](./project-overview.md)** で機能を配置すべき場所を確認
2. **[アーキテクチャ詳細ガイド](./architecture-deep-dive.md)** で類似機能の実装パターンを確認
3. **[開発ノート](./development-notes.md)** のチェックリストに従って実装
4. ベストプラクティスに沿ったコーディング

#### シナリオ 3: バグを修正する

1. **[開発ノート](./development-notes.md)** のトラブルシューティングを確認
2. よくある問題のセクションで類似の問題を探す
3. デバッグ手法を活用して原因を特定
4. 修正後、関連するテストを実行

#### シナリオ 4: パフォーマンスを改善する

1. **[開発ノート](./development-notes.md)** のパフォーマンスセクションを読む
2. **[アーキテクチャ詳細ガイド](./architecture-deep-dive.md)** でボトルネックになりやすい箇所を確認
3. プロファイリングツールを使用して測定
4. 最適化を実施

#### シナリオ 5: アーキテクチャを理解する

1. **[プロジェクト概要](./project-overview.md)** で全体像を把握
2. **[アーキテクチャ詳細ガイド](./architecture-deep-dive.md)** で詳細を学ぶ
3. 実際のコードを読んで理解を深める
4. 疑問点があれば開発チームに質問

## LLM 向けの推奨事項

### コンテキストの効率的な活用

1. **最初に読むべきドキュメント**:
   - タスクに関わらず、まず **[プロジェクト概要](./project-overview.md)** を参照

2. **タスク別の参照優先度**:

   | タスク | 1st | 2nd | 3rd |
   |--------|-----|-----|-----|
   | 一般的な質問 | プロジェクト概要 | 開発ノート | - |
   | バグ修正 | 開発ノート | プロジェクト概要 | アーキテクチャ詳細 |
   | 機能追加 | プロジェクト概要 | アーキテクチャ詳細 | 開発ノート |
   | リファクタリング | アーキテクチャ詳細 | 開発ノート | プロジェクト概要 |
   | パフォーマンス最適化 | 開発ノート | アーキテクチャ詳細 | - |

3. **ドキュメントが見つからない情報**:
   - 最新のコードを直接参照
   - 開発チームに質問
   - このドキュメントの更新を提案

### 支援時の心得

1. **プロジェクトのコンテキストを尊重**:
   - 既存のアーキテクチャパターンに従う
   - 確立されたベストプラクティスを推奨
   - 新しいパターンを導入する際は理由を説明

2. **具体的なファイルパスを提供**:
   - 「このファイルを参照してください: `browser-features/chrome/common/workspaces/index.ts`」のように具体的に

3. **段階的な説明**:
   - 複雑な概念は段階的に説明
   - 図やコード例を活用

4. **最新情報の確認**:
   - ドキュメントは定期的に更新されるとは限らない
   - 矛盾がある場合は実際のコードを確認

## ドキュメントの保守

### 更新のタイミング

このドキュメントは以下のタイミングで更新すべきです：

1. **大きなアーキテクチャ変更があった時**
2. **新しい主要機能が追加された時**
3. **重要なベストプラクティスが確立された時**
4. **よくある問題とその解決策が明らかになった時**
5. **ドキュメントの内容が古くなった時**

### 更新の責任者

- **開発チーム**: コードを変更する際にドキュメントも更新
- **レビュアー**: PR レビュー時にドキュメントの更新を確認
- **LLM**: 矛盾や古い情報を発見した際に指摘

### 貢献方法

1. このディレクトリのドキュメントを編集
2. 変更内容を PR として提出
3. レビューを受ける
4. マージ

## よくある質問

### Q1: このドキュメントは誰が書いたのですか？

A: このドキュメントは LLM (Claude) がプロジェクトを分析して作成したものです。人間の開発者によるレビューと更新を経て維持されます。

### Q2: ドキュメントの内容が実際のコードと異なる場合はどうすればいいですか？

A: 実際のコードが常に正です。ドキュメントの更新を提案してください。

### Q3: もっと詳しい情報が必要な場合は？

A: 以下の順序で情報を探してください：
1. このディレクトリの他のドキュメント
2. プロジェクト内の他の README やドキュメント
3. 実際のコード
4. 開発チームに質問

### Q4: 新しいトピックのドキュメントを追加したい場合は？

A: PR を提出してください。以下の構造を推奨します：
- 明確な目的
- 対象読者
- 具体的な内容
- コード例
- 参考リンク
